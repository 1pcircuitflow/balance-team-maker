# /review-code - 코드리뷰 어시스턴트

변경된 코드를 분석하고, 버그·보안·성능·가독성·컨벤션 관점에서 리뷰하여 구조화된 피드백을 제공합니다.

## 사용법
```
/review-code                     # 현재 스테이징/언스테이징 변경사항 전체 리뷰
/review-code <파일_또는_범위>     # 특정 파일, 커밋, 브랜치 등 지정 리뷰
/review-code #123                # GitHub PR 번호로 리뷰
```

## 인수 ($ARGUMENTS)
$ARGUMENTS

## 지침

### 1단계: 리뷰 대상 파악
1. `$ARGUMENTS`를 파싱하여 리뷰 범위를 결정합니다:
   - **인수 없음:** `git diff`와 `git diff --cached`를 실행하여 현재 변경사항을 확인합니다.
   - **파일 경로:** 해당 파일의 `git diff`를 확인합니다. diff가 없으면 파일 전체를 리뷰합니다.
   - **커밋 해시/범위:** `git diff <범위>`를 사용합니다. (예: `HEAD~3..HEAD`)
   - **브랜치명:** `git diff main...<브랜치>`를 사용합니다.
   - **PR 번호 (`#N`):** `gh pr diff N`을 실행하여 PR의 변경사항을 가져옵니다. `gh pr view N`으로 PR 설명도 함께 확인합니다.
2. 변경된 파일 목록과 변경 규모(추가/삭제 라인 수)를 파악합니다.
3. 변경이 없으면 사용자에게 알리고 종료합니다.

### 2단계: 변경 컨텍스트 이해
1. 변경된 각 파일을 Read 도구로 읽어 변경의 맥락을 파악합니다.
   - **대규모 파일 (예: `App.tsx`):** 파일 전체를 읽지 않고, diff에서 변경된 라인 번호를 기준으로 해당 함수/블록만 Read의 `offset`/`limit`으로 부분 읽기합니다. 변경된 함수의 시작과 끝을 포함하도록 전후 50줄 정도 여유를 둡니다.
2. 변경이 영향을 미치는 아키텍처 레이어를 식별합니다:
   - **Context** (`contexts/`): 전역 상태 변경
   - **Hook** (`hooks/`): 비즈니스 로직 변경
   - **Service** (`services/`): 백엔드/알고리즘 변경
   - **Component** (`components/`, `pages/`, `App.tsx`): UI 변경
   - **Type/Config** (`types.ts`, `constants.ts`): 타입/상수 변경
   - **Translation** (`translations.ts`): 다국어 변경
3. 필요시 Grep/Glob으로 관련된 다른 파일도 확인하여 사이드 이펙트를 파악합니다.

### 3단계: 코드리뷰 수행
각 변경 파일에 대해 다음 관점으로 리뷰합니다:

#### 🔴 심각 (Critical) — 반드시 수정 필요
- **버그:** 로직 오류, null/undefined 미처리, 레이스 컨디션, 무한 루프/리렌더링
- **보안:** XSS, 인젝션, 민감 정보 노출, 인증/인가 우회
- **데이터 손실:** 사용자 데이터를 잃을 수 있는 코드

#### 🟡 주의 (Warning) — 수정 권장
- **성능:** 불필요한 리렌더링, 무거운 연산, 메모이제이션 누락, N+1 쿼리
- **에러 처리:** async/await에서 try-catch 누락, Promise 에러 미처리
- **타입 안전성:** any 남용, 타입 단언(as) 과다 사용, 잘못된 타입
- **상태 관리:** stale closure, useEffect 의존성 배열 누락/과다

#### 🔵 제안 (Suggestion) — 개선하면 좋음
- **가독성:** 복잡한 로직, 불명확한 변수명, 매직 넘버
- **컨벤션:** 프로젝트 코딩 스타일과 불일치 (Tailwind 패턴, 컴포넌트 패턴 등)
- **중복:** 비슷한 코드 반복, 기존 유틸리티 미활용
- **구조:** 컴포넌트 분리, 관심사 분리 개선 가능

#### 프로젝트 특화 체크리스트
- [ ] Tailwind 다크 모드(`dark:`) 누락 여부
- [ ] 번역 키가 5개 언어(ko/en/es/ja/pt)에 모두 존재하는지
- [ ] Firebase 작업의 에러 처리
- [ ] Capacitor 플러그인 호출의 플랫폼 분기 처리
- [ ] `types.ts`와 일치하는 타입 사용
- [ ] `constants.ts`의 상수 활용 (매직 넘버 대신)

### 4단계: 변경 간 연관성 분석 (Cross-file Consistency)
여러 파일이 동시에 변경된 경우, 파일 간 일관성을 검증합니다:
1. **타입 변경 전파:** `types.ts`에서 타입이 변경되면, 해당 타입을 사용하는 모든 파일이 함께 업데이트되었는지 Grep으로 확인합니다.
2. **상수 변경 전파:** `constants.ts`의 상수가 변경되면, 참조하는 곳도 업데이트되었는지 확인합니다.
3. **번역 키 동기화:** 새로운 UI 텍스트가 추가되면 `translations.ts`에 5개 언어 모두 키가 존재하는지 확인합니다.
4. **인터페이스 계약:** Hook의 반환값이 변경되면 이를 사용하는 컴포넌트도 업데이트되었는지, Service의 파라미터가 변경되면 호출부도 수정되었는지 확인합니다.
5. **누락 감지:** 변경되어야 하는데 변경되지 않은 파일이 있다면 🟡 주의로 보고합니다.

### 5단계: TypeScript 검증
1. `npx tsc --noEmit`을 실행하여 변경사항이 타입 에러를 유발하지 않는지 확인합니다.
2. 타입 에러가 발견되면 🔴 심각으로 분류하고, 에러 메시지와 수정 방향을 함께 보고합니다.
3. 기존에 이미 존재하던 타입 에러는 별도로 구분하여 언급합니다 (이번 변경과 무관한 에러).

### 6단계: 리뷰 결과 정리
다음 형식으로 구조화된 리뷰를 출력합니다:

```
## 코드리뷰 결과

### 요약
- **변경 범위:** (변경된 파일 수, 추가/삭제 라인 수)
- **전체 평가:** (간략한 한 줄 평가)

### 🔴 심각 (N건)
**[파일명:라인]** 제목
> 해당 코드 스니펫
문제 설명 및 수정 제안

### 🟡 주의 (N건)
**[파일명:라인]** 제목
> 해당 코드 스니펫
문제 설명 및 수정 제안

### 🔵 제안 (N건)
**[파일명:라인]** 제목
설명 및 개선 제안

### ✅ 잘된 점
- 잘 작성된 부분에 대한 긍정적 피드백 (1~3건)

### 최종 의견
전반적인 코드 품질과 머지 가능 여부에 대한 의견.
```

### 7단계: 자동 수정 제안 (선택)
- 심각(Critical) 이슈가 있을 경우, 수정 코드를 함께 제안합니다.
- 단, **직접 수정은 하지 않고** 코드 블록으로 제안만 합니다.
- 사용자가 수정을 요청하면 그때 Edit 도구로 적용합니다.

## 중요 참고 사항
- 리뷰는 **변경된 코드**에 집중합니다. 기존 코드의 문제는 별도로 언급하되 주요 리뷰 대상에서 구분합니다.
- 코드를 직접 수정하지 않습니다. 리뷰어의 역할에 충실합니다.
- 사소한 스타일 이슈보다 실질적인 버그/성능/보안 문제를 우선합니다.
- 지적만 하지 말고, 잘된 점도 반드시 언급합니다.
- 수정 제안 시 프로젝트의 기존 패턴(Context + Hook, Tailwind CSS, 번역 구조)을 따릅니다.
