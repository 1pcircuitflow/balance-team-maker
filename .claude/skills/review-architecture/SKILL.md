# /review-architecture - 아키텍처 리뷰 어시스턴트

프로젝트의 구조와 설계가 합리적인지 분석하고, 모듈 분리·의존성 방향·확장성·일관성 관점에서 구조화된 피드백을 제공합니다.

## 사용법
```
/review-architecture                  # 프로젝트 전체 아키텍처 리뷰
/review-architecture <영역>           # 특정 영역만 리뷰 (예: hooks, services, contexts)
/review-architecture <파일_경로>      # 특정 파일/디렉토리 중심으로 리뷰
```

## 인수 ($ARGUMENTS)
$ARGUMENTS

## 지침

### 1단계: 리뷰 범위 결정
1. `$ARGUMENTS`를 파싱하여 리뷰 범위를 결정합니다:
   - **인수 없음:** 프로젝트 전체 아키텍처를 리뷰합니다.
   - **영역 키워드:** 해당 레이어만 집중 리뷰합니다. (예: `hooks`, `services`, `contexts`, `components`, `pages`)
   - **파일/디렉토리 경로:** 해당 파일과 연관된 아키텍처를 리뷰합니다.
2. 프로젝트의 디렉토리 구조를 Glob으로 스캔하여 전체 파일 구성을 파악합니다.
3. `package.json`을 읽어 의존성과 스크립트를 확인합니다.

### 2단계: 레이어 구조 분석
프로젝트의 아키텍처 레이어를 식별하고 각 레이어의 역할과 책임을 분석합니다:

#### 2a. 데이터 모델 레이어
- `types.ts` - 타입 정의의 완전성과 일관성
- `constants.ts` - 상수 관리의 적절성
- 타입 간 관계와 의존성이 명확한지

#### 2b. 서비스 레이어 (`services/`)
- 각 서비스의 책임이 명확히 분리되어 있는지
- 서비스 간 의존성이 적절한지
- 외부 의존성(Firebase, API)이 잘 캡슐화되어 있는지

#### 2c. 상태 관리 레이어 (`contexts/`, `hooks/`)
- Context의 범위가 적절한지 (너무 크거나 너무 작지 않은지)
- Hook이 단일 책임 원칙을 따르는지
- Context와 Hook 간 역할 분리가 명확한지
- 상태의 흐름이 예측 가능한지

#### 2d. 프레젠테이션 레이어 (`components/`, `pages/`, `App.tsx`)
- 컴포넌트 분리 기준이 일관적인지
- 페이지와 컴포넌트의 역할 구분이 명확한지
- 비즈니스 로직이 UI 레이어에 침투하지 않았는지
- 재사용 가능한 컴포넌트가 적절히 추출되어 있는지

#### 2e. 횡단 관심사
- 번역(`translations.ts`) 구조와 사용 패턴
- 에러 처리 전략의 일관성
- 플랫폼 분기(Capacitor) 처리 방식

### 3단계: 의존성 방향 검증
코드를 Grep/Read로 분석하여 의존성 방향이 올바른지 검증합니다:

```
올바른 방향:
  Component → Hook → Service → External(Firebase, API)
      ↕          ↕
   Context    balanceService
      ↕
  translations.ts

잘못된 방향 (안티패턴):
  Service → Component (서비스가 UI를 알면 안 됨)
  Context → Hook의 내부 구현 (추상화 위반)
  Component → 다른 Component의 내부 상태 직접 접근
```

1. 각 레이어의 import문을 분석하여 의존성 방향을 추적합니다.
2. 순환 의존성(circular dependency)이 있는지 확인합니다.
3. 하위 레이어가 상위 레이어를 참조하는 역방향 의존성을 찾습니다.

### 4단계: 설계 원칙 평가
다음 설계 원칙에 비추어 아키텍처를 평가합니다:

#### 🔴 심각 (Critical) — 구조적 결함
- **순환 의존성:** 모듈 간 순환 참조
- **레이어 위반:** 의존성 방향 역전
- **God Object:** 하나의 파일/모듈이 너무 많은 책임을 담당
- **단일 실패 지점:** 한 모듈의 변경이 전체 시스템에 영향

#### 🟡 주의 (Warning) — 개선 필요
- **높은 결합도:** 모듈 간 불필요하게 강한 의존
- **낮은 응집도:** 관련 없는 기능이 한 모듈에 모여 있음
- **일관성 부족:** 유사한 기능이 다른 패턴으로 구현됨
- **추상화 누락:** 반복되는 패턴이 추상화되지 않음
- **과도한 Props Drilling:** 깊은 Props 전달 체인

#### 🔵 제안 (Suggestion) — 개선하면 좋음
- **네이밍 컨벤션:** 파일/디렉토리 명명 일관성
- **디렉토리 구조:** 더 나은 구성 방법 제안
- **모듈 분리:** 큰 파일을 분리할 수 있는 지점
- **패턴 통일:** 프로젝트 전체에서 일관된 패턴 사용 제안

### 5단계: 파일 크기 및 복잡도 분석
1. 각 주요 파일의 라인 수를 확인합니다.
2. 과도하게 큰 파일(500줄 이상)을 식별합니다.
3. 복잡도가 높은 함수를 찾습니다:
   - 깊은 중첩 (3단계 이상)
   - 긴 함수 (50줄 이상)
   - 과도한 매개변수 (5개 이상)
4. 파일 분리가 필요한 지점을 구체적으로 제안합니다.

### 6단계: 확장성 평가
현재 아키텍처가 다음 시나리오에서 확장 가능한지 평가합니다:
1. **새로운 스포츠 종목 추가:** 현재 구조에서 쉽게 추가할 수 있는가?
2. **새로운 페이지/기능 추가:** 기존 패턴을 따라 쉽게 만들 수 있는가?
3. **새로운 언어 추가:** 번역 구조가 확장 가능한가?
4. **새로운 서비스 통합:** 외부 서비스를 추가하기 쉬운 구조인가?

### 7단계: 리뷰 결과 정리
다음 형식으로 구조화된 아키텍처 리뷰를 출력합니다:

- **아키텍처 성숙도:** ⭐⭐⭐☆☆ (5점 만점)
- **전체 평가:** (간략한 한 줄 평가)

## 주요 파일 참조
| 영역 | 파일 |
|------|-------|
| 메인 앱 | `App.tsx` |
| 타입/상수 | `types.ts`, `constants.ts` |
| 컨텍스트 | `contexts/AppContext.tsx`, `contexts/AuthContext.tsx`, `contexts/PlayerContext.tsx` |
| 훅 | `hooks/useTeamBalance.ts`, `hooks/useNavigation.ts`, `hooks/useRecruitmentRooms.ts`, `hooks/useInitialization.ts` |
| 서비스 | `services/balanceService.ts`, `services/firebaseService.ts`, `services/paymentService.ts`, `services/analyticsService.ts` |
| 컴포넌트 | `components/`, `pages/` |
| 번역 | `translations.ts` |
| 유틸리티 | `utils/helpers.ts` |
| 설정 | `capacitor.config.ts`, `vite.config.ts`, `tsconfig.json` |

## 아키텍처 참조
```
사용자 액션 → 컴포넌트 → 훅 → 서비스 → Firebase/API
                ↕            ↕
             컨텍스트     balanceService
                ↕
          translations.ts (i18n)
```

## 중요 참고 사항
- 리뷰는 **구조와 설계**에 집중합니다. 개별 코드 품질은 `/review-code`의 역할입니다.
- 코드를 직접 수정하지 않습니다. 리뷰어의 역할에 충실합니다.
- 현실적이고 실행 가능한 개선안을 제시합니다. 이론적으로 완벽하지만 비현실적인 제안은 피합니다.
- 프로젝트 규모와 팀 상황을 고려하여 적절한 수준의 아키텍처를 제안합니다.
- 잘된 점도 반드시 언급하여 균형 잡힌 리뷰를 제공합니다.
