# /plan-feature - 기능 기획 어시스턴트

기능 아이디어를 프로젝트의 아키텍처와 패턴을 따르는 구체적인 구현 계획으로 변환합니다.

## 사용법
```
/plan-feature <기능_설명>
```

## 인수 ($ARGUMENTS)
$ARGUMENTS

## 지침

### 1단계: 기능 이해
1. `$ARGUMENTS`에서 기능 설명을 파싱합니다.
2. 기능 범위를 명확히 합니다:
   - 어떤 문제를 해결하는가?
   - 타겟 사용자는 누구인가?
   - 기대되는 사용자 흐름은 무엇인가?

### 2단계: 기존 코드베이스 분석
1. 현재 아키텍처를 검토합니다:
   - `types.ts` - 기존 데이터 모델
   - `constants.ts` - 앱 상수 및 설정
   - `App.tsx` - 메인 앱 구조 및 라우팅
   - `contexts/` - 상태 관리 패턴
   - `hooks/` - 비즈니스 로직 패턴
   - `services/` - 백엔드 서비스 패턴
   - `components/` - UI 컴포넌트 패턴
2. 이 기능에 적용할 수 있는 기존 패턴을 식별합니다.
3. 확장 가능한 유사한 기능이 이미 존재하는지 확인합니다.

### 3단계: 구현 계획 설계
다음 섹션들을 포함하는 구조화된 계획을 생성합니다:

#### 3a. 데이터 모델 변경
- `types.ts`에 필요한 새로운 타입/인터페이스
- 기존 타입의 수정 사항
- `constants.ts`의 새로운 상수

#### 3b. 백엔드/서비스 레이어
- `services/`의 새로운 서비스 또는 기존 서비스 수정
- 필요한 Firebase 컬렉션/문서
- API 호출 또는 외부 통합

#### 3c. 상태 관리
- 필요한 경우 새로운 Context, 또는 기존 Context에 추가
- 비즈니스 로직을 위한 새로운 훅

#### 3d. UI 컴포넌트
- `components/`에 새로 만들 컴포넌트
- 수정할 기존 컴포넌트
- `pages/` 또는 `App.tsx`의 페이지 수준 변경 사항

#### 3e. 번역
- 필요한 새로운 번역 키 (모두 한국어 텍스트와 함께 나열)
- 기존 명명 규칙을 따르는 키 이름 제안

#### 3f. UI/UX 흐름
사용자 여정을 단계별로 설명합니다:
1. 진입점 (사용자가 이 기능을 어디에서 접속하는가?)
2. 상호작용 흐름
3. 성공/오류 상태
4. 화면 간 네비게이션

### 4단계: 파일 변경 요약
변경이 필요한 모든 파일의 테이블을 생성합니다:

```
| 파일 | 작업 | 설명 |
|------|--------|-------------|
| types.ts | MODIFY | NewFeatureType 인터페이스 추가 |
| components/NewFeature.tsx | CREATE | 새 컴포넌트 |
| hooks/useNewFeature.ts | CREATE | 기능 로직 훅 |
```

### 5단계: 구현 순서
의존성을 고려한 권장 구현 순서를 나열합니다:
1. 타입 및 상수 (기반)
2. 서비스 레이어 (데이터)
3. Context/Hook (상태)
4. 컴포넌트 (UI)
5. 번역 (i18n)
6. App.tsx 통합 (연동)

### 6단계: 고려 사항
- **성능:** 잠재적인 성능 관련 우려 사항이 있는가?
- **플랫폼:** Capacitor 관련 고려 사항 (Android/iOS/Web)?
- **마이그레이션:** 기존 사용자를 위한 데이터 마이그레이션이 필요한가?
- **수익화:** 이 기능이 프로(Pro) 기능이어야 하는가?

## 출력 형식
헤더, 표, 도움이 될만한 코드 스니펫을 포함하여 명확한 마크다운 형식으로 계획을 제시합니다. 계획은 추가 설명 없이도 구현이 가능할 정도로 상세해야 합니다.

## 아키텍처 참조
```
사용자 액션 → 컴포넌트 → 훅 → 서비스 → Firebase/API
                ↕            ↕
             컨텍스트     balanceService
                ↕
          translations.ts (i18n)
```

## 중요 참고 사항
- Context + Hook + Service + Component 분리 패턴을 따릅니다.
- 모든 사용자용 텍스트는 `translations.ts`에 있어야 합니다 (5개 언어).
- UI는 다크 모드를 지원해야 합니다 (`dark:` Tailwind 클래스).
- 모바일 우선 디자인(Capacitor 앱)을 고려합니다.
- 모달/오버레이를 위한 기존 Z_INDEX 상수를 확인합니다.
- 코드를 직접 작성하지 마십시오. 구현 계획만 세웁니다.
